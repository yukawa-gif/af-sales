## 確度ランクの定義

| ランク | 意味             |
| ------ | ---------------- |
| 売上   | 確定・入金済み   |
| 決定   | 受注確定・未入金 |
| A      | 高確度見込み     |
| B      | 中確度見込み     |
| C      | 低確度見込み     |
| 失注   | 失注確定         |

**ダッシュボードの「売上合計」はExcelの`=SUMIF(決定)+SUMIF(売上)`相当**（決定+売上の合算）。

---

## 粗利の計算式

```
粗利（単月）= 売上 - 費用
総粗利 = 粗利 × 月数 × コース数 × 数量
```

インポート時は**粗利を直接入力**（売上・費用は参考値）。
GASが利益を計算するのではなく、入力値の粗利をそのまま使う設計。

## 重要な技術的知見

### GAS日付型のバグ対策

スプレッドシートのDate型は `instanceof Date` で判定できない（GAS環境のバグ）。
必ずduck typingで判定する：

```javascript
typeof v.getFullYear === 'function';
```

### localStorageは使わない

以前localStorageをキャッシュに使っていたが廃止。GASスプレッドシートを唯一のデータソースとする。

### file://プロトコルは使えない

`file://`でdashboard.htmlを開くとGASへのfetchがセキュリティエラーになる。
必ずローカルサーバー経由で動作確認：

```bash
python -m http.server 8080
# → http://localhost:8080/dashboard.html
```

### GASシートの列順序は変えない

`getMaster()`と書き込み系関数で列インデックスを直接参照しているため、
列を追加する場合は末尾に追加すること。

## 技術的な学びと注意点

### GAS（Google Apps Script）連携

- `mode: 'no-cors'` でPOSTするとレスポンスが読めない → 成功/失敗の確認ができない
- GASを変更したら必ず**再デプロイ**が必要（忘れると古いコードが動く）
- シートの**列順序**は一度決めたら変えない（getMaster()と列順が食い違うとバグになる）

### localStorageの扱い

- localStorageは「キャッシュ」であって「DB」ではない
- 顧客DBのような大きなデータをlocalStorageに保存すると `QuotaExceededError` が発生する
  → `localStorage.setItem()` は必ず `try-catch` で囲む。保存失敗でもデータ取得は成功させる
- **GASを正として、同期時は完全置換する設計**が正しい（部分更新は危険）

### 商材データ

- 月数フィールド（ストック=12、ショット=1）で月別タイムラインに分配される
- インセンティブ計算は商材マスタの `incentiveType` / `incentiveTarget` で制御（変更予定）
- **正しい計算式**: 粗利（月次）= (売上単価 - 費用) × コース数 × 件数 / 総粗利 = 粗利（月次）× 月数

### Vercelデプロイ

- ローカルファイルを編集しても本番には反映されない
- `git push origin main` して初めてVercelに反映される（1〜2分かかる）
- ブラウザのキャッシュが残る場合は `Ctrl+Shift+R` で強制リロード

### 設計の反省（前回中断時）

- 出発点：「営業の目標設定・達成管理」（担当者起点）
- 途中からの実態：「顧客DB管理・担当者引き継ぎ管理」（顧客起点）
- この2つは土台が違う。顧客DBはスプレッドシートのまま運用で十分。
